<objective>
Wire the block-user feature (FEAT-05) into the three client surfaces it touches: messaging (hide convos + respect blocks in realtime), browse (best-effort app filter), and the block/unblock UI (seller profile + message thread + unblock list). The server-side enforcement (messages_block_rls.sql + is_blocked_pair) was applied in Plan 02-02 — this plan is the client wiring on top of that boundary.

Purpose: The security boundary for blocking is the messaging RLS (already live). The browse-hide is best-effort UX (Pitfall 5). Blocking is silent + mutual (D-08): the blocked party is never told, and a blocked send's RLS failure is swallowed without a "you are blocked" message.
Output: MessagesContext block-aware refresh + realtime teardown; ProductsContext block-list browse filter; Messages/Profile block controls + Profile unblock list; bilingual strings.
</objective>

| T-02-15 | Information Disclosure | leaking the block via UI ("you are blocked") | mitigate | silent UX (D-08): no block-notice copy; blocked-send RLS error swallowed client-side |
