<objective>
Wire the anti-spam controls: Turnstile CAPTCHA on signup (native Supabase Auth verification) and on posting (server-side verify before insert), plus the friendly localized surfacing of the DB-enforced per-user rate limit (ABUSE-01) when a user exceeds 10 posts/hour.

Purpose: The bypass-resistant boundaries are server-side and already live (native Auth CAPTCHA verifies the signup token inside GoTrue; enforce_post_rate_limit() raises P0001 in the DB; verify-turnstile gates posting). This plan is the client glue: render the widgets, pass the tokens, and map the coded rate-limit error to a localized message. The posting CAPTCHA is anti-bot, NOT an authorization boundary (a raw insert bypasses it — the rate-limit trigger is the backstop; document this for Phase-4 SEC-09).
Output: Register signup widget + token; AuthContext signUp captchaToken; PostAd posting widget + verify invoke + rate-limit error mapping; bilingual strings.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client signup → Supabase Auth | GoTrue verifies the Turnstile token server-side; client only renders the widget |
| client posting → Edge Function → siteverify | Anti-bot only; the DB rate-limit trigger is the authorization-grade backstop |
| client insert → products triggers | enforce_post_rate_limit() raises in the DB regardless of the client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering/DoS | client-only rate limit bypassed via raw supabase-js | mitigate | DB enforce_post_rate_limit() BEFORE INSERT (02-02) is the boundary; this plan only surfaces the coded error (HIGH) |
| T-02-19 | Spoofing | CAPTCHA token replay/reuse | mitigate | Turnstile tokens single-use; widget reset after submit; siteverify rejects reuse |
| T-02-20 | Information Disclosure | posting CAPTCHA mistaken for an authz boundary | accept (documented) | SUMMARY records it is anti-bot only; the rate-limit trigger is the backstop (Phase-4 SEC-09) |
| T-02-06 | Information Disclosure | secret key in client bundle | mitigate | only REACT_APP_TURNSTILE_SITE_KEY (public) is in the client; secret lives in the Edge Function env |
</threat_model>
