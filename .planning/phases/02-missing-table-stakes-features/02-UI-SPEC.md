# Phase 2 ŌĆö UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Brownfield phase.** CaboFeira has an existing vanilla-CSS visual language with a complete CSS-custom-property token system in `cabofeira/src/index.css`. This contract **extends** that language ŌĆö it does NOT introduce a new design system. Every new surface must reuse existing tokens (`--cf-*`), existing component shells (`.btn`, `.badge`, `.modal`, `.confirm-dialog`, `.admin-table`, `.dropdown`), and the existing bilingual i18n dictionaries. **No new CSS variables, no new font, no framework.**

## Spacing Scale

The existing codebase does NOT use a strict 8-point scale ŌĆö it uses an established practical scale derived from the real CSS. **Reuse these existing values; do not invent new spacing.** Multiples-of-4 are the baseline; the documented exceptions are pre-existing and must be matched for visual consistency, not "corrected."

| Token (conceptual) | Value | Usage (from existing CSS) |
|--------------------|-------|---------------------------|
| 2xs | 4px | `.detail-actions` gap, tag gaps, micro-margins |
| xs | 6px | `.btn` icon gap, `.card-body` gap, `.seller-name` gap |
| sm | 8px | `.action-row` gap, badge/avatar offsets, compact spacing |
| md | 12px | `.card-body` padding, `.modal-body` gap, `.admin-table` cell padding |
| base | 16px | `.btn` horizontal padding, `.confirm-backdrop`/`.modal-backdrop` padding |
| lg | 18pxŌĆō24px | `.confirm-dialog` padding (22/24), `.modal-body`/`.detail-section` padding, page margins |
| xl | 30px | `.page` top padding |
| 2xl | 48pxŌĆō60px | `.page` bottom padding, major section breaks |

**Exceptions (pre-existing ŌĆö match, don't change):**
- Badge padding `2px 8px` (`.badge`) ŌĆö sub-4 vertical is the established pill geometry; `.badge-sold` MUST match it exactly.
- `.unread-dot` 10px and `.unread-badge` `1px 7px` ŌĆö the new bell unread count must visually align with the existing `.unread-badge` (reuse the class or mirror it pixel-for-pixel).
- `.btn` padding `10px 16px`; `.action-row .btn` padding `8px 6px` (compact contact-button row) ŌĆö the WhatsApp button joins `.action-row`/contact group and MUST adopt the same compact padding as its row siblings.
- **Touch targets:** mobile bell trigger and dropdown items must be Ōēź44px tall on `max-width: 900px` (mirror the existing mobile nav-link padding `12px 10px` and search height `44px`). Desktop dropdown items reuse `.dropdown a/button` padding `10px 12px`.

## Typography

CaboFeira uses **`rem`-based sizing** (not fixed px) and a system font stack. **Reuse the existing scale below; do not introduce new sizes or a new family.** Two weights carry the whole app: 600 (semibold, the dominant emphasis weight) and 700ŌĆō800 (bold, prices/logo). 400/500 are the body/link defaults from the UA + `.navbar-links` (500). For this contract treat **600 = emphasis** and **700/800 = strong** as the two working weights; do not introduce a third custom weight.

| Role | Size | Weight | Line Height | Source / where it applies in this phase |
|------|------|--------|-------------|------------------------------------------|
| Micro / badge | 0.72rem (~11.5px) | 600 | 1 (pill) | `.badge` ŌåÆ `.badge-sold`; `.unread-badge` (bell count); audit-table uppercase `th` is 0.82rem |
| Small / meta | 0.78ŌĆō0.88rem | 400ŌĆō600 | 1.3 | notification item timestamp, report-reason helper text, audit "actor/timestamp" cells, WhatsApp button label (0.85rem in `.action-row`) |
| Body | 0.92ŌĆō0.95rem (~15px) | 400 | 1.45ŌĆō1.55 | dropdown items (0.92), modal body, confirm message (0.95, lh 1.45), notification item body |
| Label / control | 0.95rem | 600 | 1.2 | report-reason radio labels, block/unblock button text, admin-tab labels (0.95/600) |
| Sub-heading | 1.05ŌĆō1.2rem | 600ŌĆō700 | 1.2 | modal `h3` (1.05), confirm-title (1.15), admin-card `h2` (1.2), notification-dropdown header |
| Page title | 1.8rem | 700 | 1.2 | `.page-title` (unblock list lives under Profile page heading) |

**Body line-height:** 1.45ŌĆō1.55 (existing). **Heading line-height:** 1.1ŌĆō1.2 (existing). Do not override.

## Color

**Authoritative palette is `--cf-*` in `cabofeira/src/index.css`.** Reuse tokens only; do not add hex literals except where an existing badge already does (e.g. `.badge-verified` uses `#dbeafe`). The 60/30/10 split below describes the EXISTING app and the new surfaces must honor it.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--cf-bg` `#f7f8fa` + `--cf-bg-alt` `#ffffff` | Page background, card/surface/modal/dropdown backgrounds |
| Secondary (30%) | `--cf-blue` `#003893` / `--cf-blue-dark` `#002a6e` (gradient) | Navbar, primary-action emphasis, prices, active admin tab, seller avatar |
| Accent (10%) | `--cf-yellow` `#f7d116` and `--cf-accent` `#0066cc` | Reserved list below |
| Destructive | `--cf-red` `#cf2027` (badges/links) / `#dc2626` (`.btn-danger` / `.confirm` danger) | Destructive actions only |
| Success | `--cf-success` `#16a34a` | `.btn-success`, sold-confirmation success; WhatsApp uses its brand green (see note) |

**Accent reserved for (explicit ŌĆö not "all interactive elements"):**
- `--cf-yellow`: navbar CTA/"Post ad" button, search submit, active-link underline, **the notification bell unread count is RED (`#e11d48`), NOT yellow** ŌĆö it must match the existing `.unread-badge`/`.unread-dot` red so the two unread signals read as one system (D-19).
- `--cf-accent` (`#0066cc`): text links, primary buttons (`.btn-primary`), focus borders on inputs (report-reason select, search box in confirm/block dialogs).

**Per-surface color contract for new elements:**
| New element | Color | Rule |
|-------------|-------|------|
| `.badge-sold` | text on a muted/neutral fill ŌĆö use `--cf-text-muted` text on `var(--cf-border)`-family fill (mirrors base `.badge`), or a desaturated red tint. **Must NOT use `--cf-yellow`** (reserved for featured) or success green. Recommended: background `#e5e7eb` (`--cf-border`), text `#374151`; OR a clear "sold" red-tint `background:#fee2e2; color:#991b1b` to read as "no longer available." Pick one and apply identically on card + detail. | Distinct from `.badge-featured` (yellow) and `.badge-verified` (blue) |
| `.badge-verified` (extended to card + profile) | UNCHANGED ŌĆö `background:#dbeafe; color:var(--cf-blue)` | Reuse the exact existing class; only change is placement/scale |
| Notification bell icon | white (on the blue navbar) | Inherits navbar foreground |
| Bell unread count | `#e11d48` bg, white text (reuse `.unread-badge`) | Consistency with Messages badge (D-19) |
| WhatsApp button | WhatsApp brand green `#25D366` bg, white text/icon (one-off brand exception, allowed ŌĆö it is an external brand mark, like the Cape-Verde flag colors) | Sits in `.action-row` alongside contact buttons; brand-accurate green is expected by Cape Verde users |
| Block-user button / confirm | danger styling ŌĆö `.btn-danger` (`#dc2626`) inside `ConfirmDialog` with `danger` prop | Destructive-action color only |
| Unblock button | neutral `.btn-outline` (not danger ŌĆö it's a restorative action) | Not destructive |
| Report-reason selected radio | `--cf-accent` (`#0066cc`) selection ring/border | Matches existing `select:focus` accent |
| Audit-log tab (active) | `--cf-blue` underline (reuse `.admin-tabs button.is-active`) | Reuse existing admin-tab styling |

> WhatsApp green is the single sanctioned brand-color exception in this phase. No other new green/yellow may be introduced.

## Surface-by-Surface Interaction Contract

Detailed contracts for the seven UI surfaces this phase introduces/extends. Reference file/line anchors from `02-CONTEXT.md` ┬¦App integration points.

### 1. Sold badge + mark-as-sold control (FEAT-01)
- **`.badge-sold`** ŌĆö new class in `ProductCard.css`/`index.css`, geometry identical to `.badge` (`padding:2px 8px; border-radius:999px; font-size:0.72rem; font-weight:600`). On the card it occupies the same absolute slot as other badges: `.card-image-wrap .badge { position:absolute; top:10px; left:10px }`. If both featured + sold could co-occur, stack them with an 8px gap (sold wins visual priority ŌĆö it signals unavailability).
- On **ProductDetail**, render `.badge-sold` in the existing badge/tag row (`.detail-tags`) near the title.
- **Mark-as-sold control** lives per-ad in **My Ads** (`MyAds.jsx` ~L19-87) as a button next to the existing edit/delete, opening `ConfirmDialog` (mirrors the delete flow). Toggles label between "Mark as sold" / "Mark as active" based on `status`. Optimistic UI (mirror favorites pattern), async DB sync via `updateProduct({status})`.
- Sold ads are visible on the seller's **public profile ad list** with the badge, but absent from Home/Search (D-13/D-14 ŌĆö app filters `status='active'` for the feed).

### 2. Verified-seller badge (FEAT-04)
- Reuse the **exact existing `.badge-verified`** (`#dbeafe` / `--cf-blue`). No new styling.
- **ProductDetail:** already present ŌĆö no change.
- **ProductCard:** add the badge near the seller/meta line or as a small inline pill after the price; at card scale keep it to the badge's natural 0.72rem ŌĆö do NOT enlarge. If space is tight, an icon-only verified check (`FiCheckCircle` in `--cf-blue`) with `aria-label="Verified seller"` is acceptable, but the pill is preferred for consistency.
- **Profile (seller page):** render `.badge-verified` next to the seller name (`.seller-name` already lays out an inline gap of 6px ŌĆö reuse that pattern).
- Data already flows (`seller.verified` in `PRODUCT_SELECT`/`fromRow`) ŌĆö pure presentation.

### 3. Notification bell (FEAT-08)
- **Placement:** in the navbar action cluster, LEFT of the user menu (`Navbar.jsx` ~L108-151), as a button styled like `.user-trigger`/an icon button on the blue navbar (white glyph). It is **additive** to the existing Messages-link `.unread-badge` ŌĆö both coexist (D-19); the bell is the global feed, the Messages badge is per-conversation.
- **Unread count:** reuse the existing `.unread-badge` (red `#e11d48`, white text, min-width 18px) positioned on the bell glyph (mirror `.unread-dot` offset). If zero unread, show the bare bell with no badge.
- **Dropdown:** reuse the existing `.dropdown` shell (white, `--cf-shadow-lg`, `min-width:200px` ŌĆö widen to ~320px for notification rows). Header row "Notifications" (`notifications.title`). Each item: sender/title (0.92rem/600), body preview (0.88rem/400, truncated to oneŌĆōtwo lines), relative timestamp (0.78rem muted, reuse existing `timeAgo`). Whole item is a link to `/messages` (new_message) or the listing (system). Unread items get a subtle left accent or bolder weight; read items normal.
- **Mark-read on open (D-17):** opening the dropdown sets `read_at=now()` for shown unread items (optimistic; DB update via owner-scoped RLS). Also mark a `new_message` notification read when its conversation is opened (D-19).
- **Empty state:** see Copywriting ŌĆö "No notifications yet" centered in the dropdown.
- **Mobile (`max-width:900px`):** the navbar collapses to a drawer. The bell either (a) stays in the top bar outside the drawer (preferred ŌĆö notifications are glanceable), or (b) becomes a full-width drawer item. Dropdown items render at Ōēź44px touch height; on mobile the dropdown follows the existing `.dropdown` static/in-drawer treatment (white-on-translucent per the existing mobile dropdown rules).
- **Accessibility:** bell button `aria-label="Notifications ({count} unread)"`, `aria-haspopup="menu"`, `aria-expanded`; dropdown is a focus-trapped/escapable menu; items keyboard-navigable; close on Escape and outside-click (mirror the existing user-menu dropdown behavior).

### 4. Block-user controls + unblock list (FEAT-05)
- **Block control** appears (a) on the **seller Profile** page and (b) inside the **message thread** (`Messages.jsx`). Style: a discreet `.btn-outline` or text button ("Block user", `FiSlash`); on click opens `ConfirmDialog` with `danger=true` and confirm text "Block".
- **Silent + mutual (D-08):** after blocking, the conversation disappears from the Messages list (both sides) and no "blocked" notice is shown to the other party. Blocked sends fail at RLS and are swallowed client-side (NO error copy).
- **Unblock list** lives in **Profile / settings** as a simple list section: each row = avatar (reuse `.avatar-tiny`/seller avatar) + name + an "Unblock" `.btn-outline` button (non-danger). Confirm via `ConfirmDialog` (`danger=false`). Empty state per Copywriting.
- **Accessibility:** block/unblock buttons have explicit labels; confirm dialogs already focus-managed via `ConfirmDialog`.

### 5. WhatsApp contact + Share (FEAT-06, FEAT-07)
- **WhatsApp button** joins the existing contact-button group on ProductDetail (`.detail-actions` / `.action-row`, ~L254-275), adopting `.action-row .btn` compact padding (`8px 6px`, 0.85rem). Brand-green `#25D366` bg, white `FaWhatsapp` icon + "Chat on WhatsApp" label. **Rendered ONLY when `seller.phone` exists** (D-23). Links to `https://wa.me/<phone>?text=<title + listing URL>`, `target="_blank" rel="noopener noreferrer"`.
- **Share:** extend the existing `share()` (~L167-179). Where `navigator.share` exists, use the native sheet. Provide explicit fallbacks: **Share on WhatsApp** (`https://wa.me/?text=<title + URL>`, `FaWhatsapp`) and **Copy link** (`FiLink`, writes to clipboard ŌåÆ "Link copied" toast). Lay these out as a small button row reusing `.btn-outline`.

### 6. Structured report reasons (FEAT-02, FEAT-03)
- Replace the free-text/`<select>` in the existing report **modal** (`ProductDetail.jsx` ~L384-469, reuse `.modal`/`.modal-body` shell). Use a **radio-group** (preferred for Ōēż6 options, more touch-friendly than a select) OR keep the existing styled `<select>` if simpler ŌĆö either way the **stored value is the enum key** (`scam` / `spam` / `prohibited` / `already_sold` / `duplicate` / `other`), with bilingual display labels.
  | Enum value | Label (EN) | i18n key |
  |------------|-----------|----------|
  | scam | "Scam or fraud" | `report.reason.scam` |
  | spam | "Spam" | `report.reason.spam` |
  | prohibited | "Prohibited item" | `report.reason.prohibited` |
  | already_sold | "Already sold" | `report.reason.alreadySold` |
  | duplicate | "Duplicate listing" | `report.reason.duplicate` |
  | other | "Other" | `report.reason.other` |
- Optional free-text **details** textarea (reuse `.modal-body textarea`), shown always or when "Other" is selected.
- Selected radio uses `--cf-accent` ring (match `select:focus`). Submit button = `.btn-primary` "Submit report".
- **Dedup is silent (D-21):** a repeat report on the same listing succeeds silently (no error). On already-reported, show a calm confirmation/disabled state ("You've already reported this listing", `report.alreadyReported`) rather than an error ŌĆö but never a hard error.

### 7. Admin audit-log tab + reports reason filter (ADMIN-02, FEAT-02)
- **New "Audit log" tab** in `Admin.jsx` tab bar (reuse `.admin-tabs button` / `.is-active` blue-underline styling, alongside users/ads/reports/pricing).
- **Read-only table** reusing `.admin-table` (uppercase muted `th`, hover rows, `.table-wrap` horizontal scroll). Columns: **Action ┬Ę Target ┬Ę Actor ┬Ę Timestamp** (use `timeAgo`/locale format consistent with the rest of admin). Column header keys: `admin.audit.action` / `.target` / `.actor` / `.time`.
- **Filter control** (by action; reports tab gets a by-reason filter): a `<select>` styled to match existing admin filters in the `.row-between` header row. Reason filter options reuse the same six `report.reason.*` labels plus "All reasons" (`admin.reportsAllReasons`).
- **Pagination** for volume: reuse whatever pagination pattern the admin/Search lists already use (server-side `range`, consistent button styling). If none exists in Admin yet, use simple Prev/Next `.btn-outline` controls.
- Empty/filtered-empty states per Copywriting.
- Read-only: no row actions; RLS already restricts SELECT to admins.
