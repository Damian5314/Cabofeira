<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Postgres | Authenticated user may read/mark-read only own notifications; no direct insert. |
| realtime channel → subscriber | Postgres-changes stream filtered by RLS SELECT policy. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | reading others' notifications (direct or via realtime) | mitigate | owner-only `notifications_select_own`; RLS-filtered postgres_changes delivers only owner rows (D-10, FND-02) |
| T-02-02 | Spoofing/Tampering | forging notifications for other users via direct INSERT | mitigate | no direct INSERT policy; sole path is create_notification() SECURITY DEFINER (D-06/D-10) |
| T-02-03 | Tampering | mark-read update rewriting type/title to another user's row | mitigate | update policy `with check (auth.uid()=user_id)` keeps row owned; cross-user update denied |
| T-02-04 | Elevation of Privilege | search_path hijack on create_notification | mitigate | `set search_path = public`; execute revoked from anon/public |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (SQL only); audit table N/A per RESEARCH |
</threat_model>
